## Incident: 2026-04-07 12:00 UTC

## Summary

Around 12:00 UTC on April 7, 2026, the JWT Pizza service stopped successfully processing orders. Every attempt to place an order resulted in a 500 error on the /api/order endpoint, and users were shown a failure message in the UI.

The issue was caused by an active chaos condition in the Pizza Factory service, which forced order failures. The incident lasted for about an hour and a half until it was resolved by following a remediation link found in the API response.

This was a high severity incident because it completely blocked the main functionality of the application, which is placing orders.

---

## Detection

The incident was detected through multiple signals in Grafana and the application:

- HTTP 500 Errors alert fired in Grafana
- Factory 500 Errors alert began firing repeatedly
- The frontend displayed "Failed to fulfill order at factory"

In the logs, I could see entries like:

POST /api/order returning statusCode 500  
Factory -> 500 success false  
resBody message "{\"message\":\"Failed to fulfill order at factory\",\"followLinkToEndChaos\":\"https://pizza-factory.cs329.click/api/support/cba92b92c7224824aae449771ee072c9/report/apyt3fhp\"}"

The alerts worked well and triggered quickly once failures started happening.

One improvement would be making alerts more specific about downstream failures instead of just generic 500 errors so the root cause is clearer faster.

The alerts and logs together made it clear that the failure was coming from the factory service rather than the application itself.

---

## Impact

For about 1.5 hours, users were unable to place any orders.

- 100 percent of order attempts failed
- Users saw error messages during checkout
- The application was still running but the main feature was broken
- No orders could be completed during this time

There was no evidence of data corruption or data loss.

## Response

The response started by checking Grafana dashboards and alerts.

I looked at:

- HTTP Request Logs panel
- Factory Service Logs panel
- Unhandled Exception Logs panel

From there I noticed that every failed request had a response body that included a followLinkToEndChaos URL.

That pointed directly to the remediation path. After opening that link in the browser, the factory service switched from chaotic to calm.

The main delay was figuring out that the fix was inside the response body and not in the application or database.

---

## Root Cause

The root cause was an active chaos injection in the Pizza Factory service.

This chaos condition forced the factory to return 500 errors for order requests. The backend passed these failures directly to the frontend, which caused all order attempts to fail.

The system does not currently have any retry or fallback behavior, so failures in the factory immediately break the full order flow.

---

## Resolution

The issue was resolved by using the followLinkToEndChaos URL found in the API response.

After navigating to that link, the factory service exited chaos mode and returned to normal operation.

Verification steps included:

- Successfully placing an order through the UI
- Confirming factory logs showed success instead of 500 errors
- Confirming HTTP 500 alerts stopped firing
- Confirming alerts resolved in Grafana

---

## Prevention

This incident showed that the system depends heavily on the factory service and has no protection against downstream failure.

Similar issues could happen if the factory service goes down or becomes unstable in the future.

The system should also be better protected against malformed or malicious requests. During testing, there were multiple unexpected requests hitting the service, which caused parsing errors and noise in the logs. Adding better request validation and filtering could help prevent these from triggering unnecessary errors or alerts.

---

## Action Items

1. Add retry logic for factory requests to handle temporary failures
2. Improve frontend messaging so users see a clearer explanation when orders fail
3. Improve alert descriptions to clearly identify factory related failures
4. Surface the chaos remediation link more clearly instead of hiding it in logs
5. Document the chaos recovery process so future incidents can be resolved faster
6. Add request validation and filtering to handle malformed or malicious requests and reduce unnecessary errors and alert noise
